## 阶段一：基础环境

### [2026-03-09] setup.sh 运行报错退出（Exit Code 1）

**现象**: 执行 `./scripts/setup.sh` 后立即退出，未启动任何服务。

**原因**: `setup.sh` 会校验 `.env` 中的关键变量，以下任一条件触发退出：
- `N8N_PASSWORD` 仍为默认值 `your_strong_password_here`
- `DEEPSEEK_API_KEY` 仍为 `sk-xxxxxxxx...` 占位符
- `.env` 文件不存在（首次运行自动复制后需重新执行）

**解决方案**:
```bash
# 1. 编辑 .env，填写真实值
nano project/.env

# 2. 生成一个强密码和加密密钥（复制输出结果填入 .env）
openssl rand -base64 16   # 用作 N8N_PASSWORD
openssl rand -hex 32      # 用作 N8N_ENCRYPTION_KEY

# 3. 重新执行安装脚本
cd project && ./scripts/setup.sh
```

**状态**: 已解决

## 阶段二：Docker 服务启动

### [2026-03-09] ChromaDB 健康检查持续 unhealthy

**现象**: `docker compose ps` 中 chromadb 状态为 `unhealthy`，paper-api 无法启动。

**排查步骤**:
```bash
# 查看详细日志
docker compose logs chromadb

# 检查端口冲突（8000 被本机其他进程占用）
lsof -i :8000
# 若有占用，修改 docker-compose.yml 中 chromadb 的端口映射，如改为 18000:8000
```

**根本原因（已修复）**:
- `chromadb/chroma:latest` 镜像内**没有 `curl`**，原健康检查 `CMD curl http://localhost:8000/api/v1/heartbeat` 直接报 `executable file not found`
- Volume 挂载路径错误：镜像实际存储于 `/data`，而不是 `/chroma/chroma`，导致数据不持久化

**修复方案**（已应用到 docker-compose.yml）:
```yaml
# 1. 健康检查改用 bash /dev/tcp（镜像内有 bash）
healthcheck:
  test: ["CMD", "bash", "-c", "echo > /dev/tcp/localhost/8000"]
  start_period: 15s   # 给 ChromaDB 更多启动时间

# 2. Volume 挂载路径修正
volumes:
  - chroma_data:/data   # 原来错误的是 /chroma/chroma
```

**状态**: ✅ 已解决

---

### [2026-03-09] Paper API 启动时报 "Connection refused chromadb:8000"

**现象**: paper-api 日志显示连接 ChromaDB 失败，服务健康检查不通过。

**原因 A**: docker-compose 的 `depends_on: condition: service_healthy` 偶尔在低配机器上因 ChromaDB 初始化慢而竞争失败。

**原因 B（已修复）**: ChromaDB Python 客户端（`chromadb==0.5.15`）与 `chromadb/chroma:latest` Docker 镜像版本不兼容。`latest` 已升级至 v2 API，Python 客户端仍使用 v1 API，启动时报：
```
The v1 API is deprecated. Please use /v2 apis
ValueError: Could not connect to tenant default_tenant
```

**修复方案**（已应用）:
```yaml
# docker-compose.yml：锁定 ChromaDB 镜像版本与 Python 客户端一致
image: chromadb/chroma:0.5.23

# requirements.txt：同步更新
chromadb==0.5.23
```

如只是重启竞争问题：
```bash
docker compose ps          # 确认 chromadb 状态为 healthy
docker compose restart paper-api
```

**状态**: ✅ 已解决（版本锁定）

### PubMed 搜索 HTTP 429 速率限制

**现象**: 多个关键词并发搜索时，PubMed 返回 429 Too Many Requests。

**原因**: 未配置 NCBI API Key 时，每个 IP 每秒限制 3 次请求；并发数 `concurrency: 3` 加上多关键词容易超限。

**解决方案**:
```bash
# 申请免费 NCBI API Key（速率提升到 10次/秒）
# 网址: https://www.ncbi.nlm.nih.gov/account/
# 申请后在 .env 中填写：
NCBI_API_KEY=your_ncbi_api_key
```
或临时调低并发数：
```yaml
# config.yaml
search:
  concurrency: 2
  request_delay: 2.0
```

**状态**: 已记录

---

### Semantic Scholar 部分论文无摘要

**现象**: 搜索结果中 `abstract` 字段为空字符串。

**原因**: Semantic Scholar 数据库收录了大量无摘要的会议论文/技术报告。

**解决方案**: `config.yaml` 中 `require_abstract: true`（已默认启用）可自动过滤。如需保留无摘要论文（仅用标题判断相关性），改为 `false`。

**状态**: 已通过配置规避

## 阶段四：AI 总结

### DeepSeek API 调用超时

**现象**: AI 总结阶段报 `ReadTimeout` 或 `openai.APITimeoutError`。

**原因**:
- 单批次论文数过多（每批 5 篇 × 摘要 2000 tokens = 约 10K tokens/批），在高负载时段响应慢
- DeepSeek 平台偶发性延迟

**解决方案**:
```yaml
# config.yaml 中调小批次
summarizer:
  batch_size: 3
  max_tokens: 400
```

**状态**: 已记录

---

## 阶段五：向量存储

### sentence-transformers 模型首次下载缓慢或失败

**现象**: Paper API 首次启动时长时间停在 "Downloading model..."，或因网络超时报错。

**原因**: 模型需从 HuggingFace Hub 下载（约 418MB），国内服务器访问慢。

**解决方案 A（推荐）：构建时预下载，写入镜像**

在 `project/Dockerfile` 中，在 `COPY` 指令后追加：
```dockerfile
# 构建镜像时预下载 Embedding 模型（约增加 418MB 镜像体积）
ENV HF_ENDPOINT=https://hf-mirror.com
RUN python3 -c "from sentence_transformers import SentenceTransformer; \
    SentenceTransformer('paraphrase-multilingual-MiniLM-L12-v2')"
```

**解决方案 B：换用更小的英文模型**（摘要以英文为主时适用）
```yaml
# config.yaml
vector_store:
  embedding_model: "all-MiniLM-L6-v2"   # 91MB，英文专用，速度更快
```

**状态**: 已记录

### 去重误判（应保留的论文被过滤）

**现象**: 同一研究方向下两篇内容不同的论文被判定为重复而丢弃。

**原因**: `dedup_similarity_threshold` 设置过低，或搜索词本身导致返回高度相似的综述类论文。

**调整方法**:
```yaml
# config.yaml 中提高阈值（更宽松）
vector_store:
  dedup_similarity_threshold: 0.97
```

**验证**（检查相似度分布）:
```bash
curl -X POST http://localhost:8080/api/search/semantic \
  -H "Content-Type: application/json" \
  -d '{"query": "你的关键词", "top_k": 5}'
# 查看返回的 similarity_score，据此调整阈值
```

**状态**: 已记录

## 阶段六：推送渠道

### Outlook 邮件配置

> **当前启用渠道**: ✅ Outlook Email + ✅ GitHub Issues

#### 配置步骤（记录备查）

**1. 确认 Outlook 账号类型**

| 账号类型 | SMTP 地址 | 端口 |
|---------|-----------|------|
| 个人 outlook.com / hotmail.com / live.com | `smtp-mail.outlook.com` | 587 |
| 企业/学校 Office 365 | `smtp.office365.com` | 587 |

两种类型均使用 **STARTTLS**（端口 587），不是 SSL 465。

**2. 是否开启了两步验证（MFA）**

- **未开启 MFA**：直接使用 Outlook 登录密码
- **已开启 MFA**：必须使用「应用密码」，否则 SMTP 认证会被拒绝

  **获取应用密码**：
  1. 访问 [account.microsoft.com](https://account.microsoft.com)
  2. 安全 → 高级安全选项 → 应用密码 → 创建新应用密码
  3. 复制生成的 16 位密码，填入 `.env` 的 `EMAIL_SMTP_PASSWORD`

**3. 确认 `.env` 配置正确**
```env
EMAIL_SMTP_HOST=smtp-mail.outlook.com
EMAIL_SMTP_PORT=587
EMAIL_SMTP_USER=your_email@outlook.com
EMAIL_SMTP_PASSWORD=你的登录密码或应用密码
EMAIL_FROM=your_email@outlook.com
EMAIL_TO=your_email@outlook.com
```

**4. 测试连接（在服务器上执行）**
```python
import smtplib
with smtplib.SMTP('smtp-mail.outlook.com', 587) as s:
    s.starttls()
    s.login('your_email@outlook.com', 'your_password')
    print("✅ Outlook SMTP 连接成功")
```

#### 常见错误

| 错误信息 | 原因 | 解决方法 |
|---------|------|---------|
| `535 5.7.139 Authentication unsuccessful` | 密码错误，或 MFA 账号未用应用密码 | 重新获取应用密码 |
| `534 5.7.9 SMTP AUTH not supported` | 账号未启用 SMTP AUTH | 前往 Microsoft 365 管理中心启用 |
| `Connection refused` | 端口错误（不要用 465） | 确认端口为 587 |
| `SMTPConnectError` | 服务器防火墙未开放 587 端口 | 在云服务商安全组放行 587 出站 |

### GitHub Issues 归档配置

#### 配置步骤（记录备查）

**1. 归档目标**

直接使用当前仓库 `haha2205/AutoReadPaper`（Issues 已启用）：
```env
GITHUB_REPO_OWNER=haha2205
GITHUB_REPO_NAME=AutoReadPaper
```

**2. 生成 Personal Access Token (PAT)**

1. 登录 GitHub → 头像 → **Settings**
2. 左下角 **Developer settings** → **Personal access tokens** → **Tokens (classic)**
3. 点击 **Generate new token (classic)**
4. 填写备注（如 `AutoReadPaper`），有效期建议选 **No expiration** 或 1 年
5. 权限勾选 **`repo`**（包含 Issues 写入）
6. 点击 **Generate token**，**立即复制**（只显示一次）
7. 填入 `.env`：
   ```env
   GITHUB_TOKEN=ghp_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
   ```

**3. 创建 paper-digest Label（首次设置）**

Issues 会自动打上 `paper-digest` 标签，需提前在仓库创建：
1. 进入 `github.com/haha2205/AutoReadPaper/issues/labels`
2. 点击 **New label**
3. 名称: `paper-digest`，颜色: `#0075ca`

**4. 测试接口**
```bash
# 手动测试创建 Issue（替换 TOKEN）
curl -X POST https://api.github.com/repos/haha2205/AutoReadPaper/issues \
  -H "Authorization: token ghp_your_token" \
  -H "Accept: application/vnd.github.v3+json" \
  -d '{"title": "测试 - 论文归档", "body": "测试内容", "labels": ["paper-digest"]}'
```
返回 HTTP 201 则配置成功。

#### 常见错误

| 错误信息 | 原因 | 解决方法 |
|---------|------|---------|
| `401 Unauthorized` | Token 错误或已过期 | 重新生成 PAT |
| `403 Forbidden` | Token 缺少 `repo` 权限 | 重新生成，勾选 repo 权限 |
| `404 Not Found` | 仓库名拼写错误 | 检查 `GITHUB_REPO_OWNER` 和 `GITHUB_REPO_NAME` |
| `422 Unprocessable Entity` | Label 不存在 | 在仓库 Labels 页面创建 `paper-digest` |

## 阶段七：n8n 工作流

### n8n 无法通过内网访问 Paper API

**现象**: n8n 工作流中 HTTP Request 节点报 `ECONNREFUSED http://paper-api:8080`

**原因**: n8n 容器和 paper-api 容器不在同一 Docker 网络中（常见于手动 `docker run` 启动 n8n 而非使用 docker-compose）。

**解决方案**: 统一使用 `docker-compose.yml` 启动所有服务，确保共享 `autoreadpaper_net` 网络。

**快速验证**:
```bash
# 进入 n8n 容器，测试能否访问 paper-api
docker exec -it autoreadpaper_n8n curl http://paper-api:8080/health
```

**状态**: 已记录

### n8n Schedule Trigger 时区不对

**现象**: 设置每天 09:00 触发，但实际执行时间偏差 8 小时（按 UTC 执行）。

**原因**: n8n 容器未设置时区。

**已在 docker-compose.yml 中配置**（无需手动处理）:
```yaml
environment:
  - TZ=Asia/Shanghai
  - GENERIC_TIMEZONE=Asia/Shanghai
```

若仍有偏差，进入 n8n 设置页面确认：Settings → **Timezone** → 选择 `Asia/Shanghai`。

### [2026-03-09] docker compose up -d n8n 卡住等待 paper-api

**现象**: `docker compose up -d n8n` 长时间卡在 `Container autoreadpaper_api  Wait...`，无法启动 n8n。

**原因**: n8n 配置了 `depends_on: paper-api: condition: service_healthy`，而 paper-api 需要从 Dockerfile 构建（含 torch、sentence-transformers 等大依赖），在 dev container 中构建耗时 5 分钟以上，period 内未通过健康检查导致 n8n 一直等待。

**修复方案**（已应用到 docker-compose.yml）:
```yaml
# 将 n8n 对 paper-api 的依赖从 service_healthy 改为 service_started
n8n:
  depends_on:
    chromadb:
      condition: service_healthy
    paper-api:
      condition: service_started   # 容器启动即可，不等健康检查
```

n8n 需要调用 paper-api HTTP 接口时，paper-api 已经完成构建并就绪，不影响工作流运行。
